### Appendix A: Architectural Recommendations on "Content as Data"
The realization that the source `.md` files are intended to act as **database entries** (localizing claims, implications, and literature history) fundamentally shifts the interpretation of "editorial bloat" originally found in F-6. Returning to monolithic files would be a disaster for tracking dependencies and preventing drift. The friction arises because the file currently acts as both the database *and* the presentation layer without strict schema boundaries.

I recommend the following staged architectural fixes:

**1. The Immediate Fix: Update the Auditor Instructions**
Currently, a de novo auditor reads top-to-bottom. The `de-novo-audit-instructions.md` must explicitly teach the auditor how to parse the schema:
> *Parsing Segment Files:* Treat `.md` files in `src/` as database entries. Your primary focus is `Formal Expression` and `Epistemic Status`. You should read `Discussion` to understand implications. **You MUST ignore `Findings`, `Search Logs`, and `Working Notes`** unless you are specifically tasked with auditing literature coverage. Do not penalize a file for having a long `Findings` block; that is working data for the build system.

**2. The Lightweight Fix: Strict Schema Boundaries**
If everything remains in one file, enforce a stronger visual and semantic boundary in `FORMAT.md`. For example, mandate that all meta-content goes below a horizontal rule (`---`) followed by a standard `# Meta & Provenance` header. This allows human readers to effortlessly filter the file visually and makes writing `bin/` extraction scripts trivial.

**3. The Heavy Fix: The "Sidecar" Pattern**
If the database entries become unreadable, split the data by concern into parallel directories (e.g., `01-aad-core/src/der-orient-cascade.md` for pure theory, and `01-aad-core/meta/der-orient-cascade.meta.md` for Findings/Logs). They remain bound by their slug, preventing monolithic drift, but keeping the `src/` directory pristine.

**4. The Ideal End-State: Lean heavily into `bin/` build artifacts**
Humans and auditors shouldn't read the raw database entries linearly. If a `build/AAD-Pedagogical-Readthrough.md` is automatically generated that strips out all `Findings` and `Working Notes`, auditors and new human contributors should only ever read the compiled views.
